# Retro — Lab Phase 1

Date: 2026-05-25 (Phase 1 shipped) / retro written 2026-06-13
Phase: plan 0007 Phase 1
Status: complete; Lab parked as of scope reset 2026-06-03
Prior doc: `plans/0007-highsignal-lab-substrate.md`

---

## What was the goal?

Ship a local-first Postgres+pgvector substrate that gives the operator a ranked,
searchable, daily feed of tech/startup documents — HN stories, linked pages, GitHub
repos — with optional semantic search, entity tagging, and story clustering. The
single success metric from plan 0007: "The operator chooses to open the feed
daily."

---

## What shipped

- docker-compose Postgres + pgvector + pg_trgm
- Schema: `documents`, `repos`, `hn_threads`, `entities`, `document_entities`, `links`, `ingest_runs`
- HN ingest with outbound-link extraction (lxml, noise filter)
- One-hop materialization via Trafilatura
- GitHub trending HTML scraper (no API key; daily/weekly/monthly × 5 languages)
- 4-factor scorer: HN discussion + recency + velocity + GitHub momentum
- Union-find story clustering (shared links + embedding cosine ≥ 0.85)
- Local sentence-transformer embeddings (MiniLM-L6-v2, 384-dim, HNSW index)
- GLiNER entity extraction (optional `[entities]` extra)
- Local-LLM summarization (Ollama / vLLM, any OpenAI-compatible endpoint)
- FastAPI server: `/feed`, `/search`, `/stats`, `/healthz`
- Web `/lab` page with cluster-collapse toggle and `LAB_API_URL` fallback

---

## What did not ship from the plan

- 14k-repo GitHub DB import — only trending HTML scrape landed
- GitHub API enrichment (stars, forks, last commit, topics) for repos
- Referenced papers/blog ingest beyond the one-hop pass
- Entity-momentum overlay on web `/entities` view
- Repo embeddings populated end-to-end (embedder supports it; no scheduled trigger)

---

## What went well

**The pipeline degrades gracefully.** Each step is independently runnable and
idempotent. The `[embeddings]` and `[entities]` extras are genuinely optional —
the feed works with just FTS + 4-factor scoring. This made it possible to ship
incrementally without a "big bang" bring-up requirement.

**Velocity as a 4th factor was the right call.** Plan 0007 adopted it from the
Digg AI competitor review (2026-05-23). Recency ≠ velocity; the rate at which a
story accumulates inbound links surfaces early-breaking stories, not just recent
ones.

**Union-find clustering is simple and effective.** The shared-link-target heuristic
requires no ML to work. The embedding-cosine layer adds quality when embeddings
are available. The fallback to link-only is correct behavior.

---

## What was harder than expected

**Phase 1 shipped code but not a running service.** The code was committed but
Postgres was not running; bring-up was on the operator. agents.md explicitly
called this out as a known gap. The Lab was never actually opened daily — the
plan's own success metric was not met.

**Optional steps had hidden dependencies.** Clustering silently falls back to
link-only without embeddings; the runbook did not make this dependency explicit.
The embed step downloads ~90 MB on first run, which was not documented.

**The 14k-repo import never landed.** This was the "spine" of the plan — without
it, `repos` only contained trending scrape results (a few dozen rows), not the
broad repo universe that would make semantic search meaningful.

---

## Why it was parked

Scope reset 2026-06-03 (`docs/scope-reset-2026-06-03.md`): "Lab is a substrate
experiment, not a product dependency." The Lab's success metric (operator opens
the feed daily) was not demonstrably met. The active brief product did not depend
on it. Keeping it as optional local tooling is correct; it should not drive
roadmap effort until the basic feed is genuinely useful to the operator.

---

## Carry-forward for if/when Lab is un-parked

1. Prove the bring-up sequence works end-to-end before calling Phase 1 done.
2. Seed the 14k-repo import before claiming the semantic-search story is real.
3. Add a bring-up health check that confirms embeddings, entities, and
   summarization are populated, not just that the server is responding.
4. Make the success metric measurable: log how often the operator opens `/lab`.
